## 2. Roadmap and Upcoming Use Cases
*Agent: Break down here the technical steps (ports, use cases, adapters, beans) for the requirements that I will request.*
**Important:** client entity is an extension of a person entity. To save a client, the person entity must be saved first.
**Entity Description:**
- Person entity attributes:
  - Person_id: primary key, autogenerated by the database.
  - first_name: the first name of the person.
  - last_name: the last name of the person.
  - age: age of the person.
  - gender_id: a foreign key to the gender entity, so far, exists just: Femenino (1), Masculino (2).
  - identification_number: unique identifier for the person, such as a passport number or national identification number.
  - address: residence address of the person.
  - postal_code: postal code of the person's residence.
  - contact_number: phone number of the person.
  - identification_type_id: a foreign key to the identification type entity, so far, exists just: DNI (1), NIF (2), PPT (3).
  - email: email address of the person. there can be null value.
- Client entity attributes:
  - Client_id: primary key, autogenerated by the database.
  - person_id: a foreign key to the person entity.
  - client_code: unique identifier for the client, such as a customer number or account number, sent in the data request.
  - password: password for the client account. it mus be saved encrypted.
  - status: status of the client account, true if active, false if inactive.
    Always can review in infrastructure>src>main>resources for the schema.sql file to see the current database structure, also can review data.sql file to see the current database data.
